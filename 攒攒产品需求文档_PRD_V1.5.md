# 一、产品定义

## 1.1 产品名称

# 攒攒

代码与仓库标识统一使用 `cuancuan`，含义来自“攒局”；不得写作 `zanzan`。中文产品名保持“攒攒”。

母品牌主张：

> **基于真实意图，帮你找到合适的人并攒成一次机会。**

核心介绍：

> 攒攒提供统一账号、Profile、Intent、Opportunity、Agent 工具、匹配能力、数据库和管理后台；用户只需进入同一个对话，根据当下需求选择合作伙伴、玩伴、相亲、招聘、创投、旅友或活动名单配对。系统在后台切换相应的访谈逻辑、候选池、结果形式和服务语言。

当前产品结构：

```text
攒攒统一对话
├── 找合作伙伴
├── 找玩伴
├── 认真相亲
├── 招聘求职
├── 找创投
├── 找旅友
└── 活动名单配对（现场攒攒）
```

## 1.2 产品本质

攒攒不是资料卡浏览平台，也不通过一套固定问卷完成所有匹配。用户进入同一个聊天工作台，可以直接表达，也可以先点选功能降低表达成本。功能选择只决定 Agent 如何继续理解，不把用户带到另一个产品或另一套登录页。

产品核心流程：

```text
用户进入统一 `/chat` 工作台
→ 直接表达，或在输入框下选择一个功能
→ 系统启用对应的专业 Agent 能力
→ Agent 结合完整上下文理解目标、处境、价值、投入方式和边界
→ 生成一版可修改的当前意图
→ 用户确认意图
→ Agent 在当前关系空间调用搜索与详情工具读取真实开放局
→ 分别检查目标一致、能力互补、城市时间、投入方式、硬性边界和多人组合缺口
→ 返回引用具体事实的个性化推荐理由与低风险下一步
→ 没有合适机会时创建新局
→ 按每个局的规则直接加入、等待发起人审核或进入候补
→ 局内推进成立、预约、执行、反馈与后续连接
→ 信用与治理数据进入真实数据库和管理后台
```

## 1.3 当前阶段验证目标

1. 用户是否愿意用自然对话表达真实关系需求；
2. Agent 是否能从非结构化表达中形成可用意图；
3. 统一入口下的功能选项是否能让用户快速理解能力范围，同时减少选择压力；
4. 推荐理由是否足以支持用户判断是否加入；
5. 用户是否愿意确认意图、加入已有局或创建新局；
6. 六类专业能力是否能在共用入口的同时保持准确的访谈逻辑与服务结果；
7. 哪类意图更容易产生有效匹配和真实参与；
8. 活动主办方能否通过一份签到表快速发现需求—资源互补关系，并在现场完成更多有效引荐。

# 二、当前版本边界

## 2.1 当前版本必须支持

* 邮箱注册和登录；
* 邀请内测码校验与限量管理；
* 根路径 `/` 为攒攒产品 Landing Page，先解释母品牌、工作方式和六类使用场景，再统一进入 `/chat`；
* Landing Page 使用用户能直接复述的生活语言：说清“想找谁、什么时候、一起做什么”，不使用“意图匹配、相容、机会生成”等内部产品术语作为主文案；
* Landing Page 首屏只保留居中的一个标题、一句说明和一个“开始和攒攒聊”主动作；首屏下方用一块大幅产品展示承接原有“不同需求的人来到同一张桌前”插画；
* Landing Page 顶部参考 Manus Download 的克制导航，只保留品牌、两个页内锚点和一个“开始使用”按钮，不增加营销菜单；
* 六个场景说明不可伪装成六个独立产品入口；依次展示合作、玩伴、相亲、招聘、创投、旅友的名称、一句用途和场景插画；
* 用户侧唯一产品入口为 `/chat`；旧 `/founder`、`/play`、`/love`、`/jobs`、`/capital`、`/travel` 链接统一重定向到 `/chat`；
* 对话框下提供“找合作伙伴、找玩伴、认真相亲、招聘求职、找创投、找旅友、活动名单配对”七个轻量功能；
* 选择功能后不换页、不重新登录，只更新当前 Agent 提示词、输入提示和后续轻量表单；
* 当前 Demo 以 Channel 配置复用同一套前端与 Agent 编排，不复制多份代码；
* 用户与 Agent 进行开放式自然对话；
* Agent 结合历史上下文动态追问，每次只追问一个关键缺口；
* 当 1–3 个选择或短回答明显比开放式表达更省力时，Agent 可调用 `ask_user_question` 展示轻量表单；
* 一次输入包含足够信息时直接生成意图，不强制走固定轮数；
* 对话理解由服务端 Vercel AI SDK `ToolLoopAgent` 编排，并调用阶跃星辰文本模型；浏览器端不得接触模型密钥；
* 模型未配置、超时或上游异常时切换到可识别的本地演示逻辑，保证主流程可继续体验；
* 活动主办方可导入当前 17 列 Excel 签到表，单次分析 2–40 位嘉宾；
* 活动页展示嘉宾完整姓名；姓名仅留在浏览器页面，发给模型的匹配资料使用匿名 ID；
* 微信、手机号等联系方式在浏览器解析阶段即剔除，不发送给模型、不写入仓库；
* 服务端先筛选可解释候选关系，再由阶跃星辰 Agent 判断最高价值的现场引荐，并用确定性规则补齐关系图；
* 对接图优先保证参与覆盖率：每位嘉宾在存在真实候选关系时都应尽可能连接到至少一位其他嘉宾；不得为了消除孤立节点虚构资源或合作理由；
* 对接图支持节点/连线查看、优先关系和双向互补筛选、现场开场建议与“已引荐”标记；
* 自动保存 Profile 和当前 Intent；
* Intent 可继续补充、纠正、重新整理；
* 用户确认 Intent 后开始匹配；
* 统一发现页可浏览全部开放机会，并按合作、玩伴、相亲、招聘、创投和旅行筛选；Agent 推荐仍只使用当前意图对应的专业候选池；
* 发现页按未来七日、城市、场景、人数状态以及推荐 / 时间 / 距离筛选活动；
* 活动列表明确展示日期、时间、地点、距离、价格、报名人数、容量和报名状态；
* 活动详情展示完整时间地点、发起人、参加者全名与摘要、当天流程、须知、报名和取消截止时间；
* Agent 优先推荐与当前 Intent 类型相容的局；
* Agent 创建新局；
* 用户提交报名、取消报名，并看到待确认、已确认或候补状态；
* 已确认用户可在详情页看到活动方配置的微信 / 企业微信群入口；
* 查看局详情、成员摘要、人数和状态；
* 查看自己加入或创建的局；
* 编辑基础资料、暂停和恢复匹配；
* 对话框下的功能选择和账号菜单等可见入口必须有真实行为；
* Agent 必须先使用 `search_rooms` 搜索当前关系空间的真实数据库，再用 `inspect_room` 查看候选局；不得只按场景标签返回固定卡片；
* 推荐解释必须按当前用户的 Intent、当前空间资料、候选成员资源、城市时间和边界逐项生成，并标记 Agent 或本地可解释降级来源；
* 每个局可独立配置“需要发起人审核 / 有名额直接加入”和“公开可发现 / 仅受邀可见”；发起人可在局详情处理申请；
* 候补按报名时间排序；已确认成员取消后自动补入最早候补，并重新计算成局状态；
* 局必须保存并展示 `招募中 → 待确认 → 已成立 → 已预约 → 进行中 → 已完成 / 已取消 → 后续连接`；
* 发起人可更新约定时间、地点、会议链接、小目标、角色分工、截止时间、完成标准和继续关系决定；
* 已确认成员可使用站内局内群聊；微信 / 企业微信群保留为可选外部承接，不再是唯一沟通方式；
* 局详情提供会前提醒、已确认成员摘要、会后到场/结果/继续意愿反馈和下一步关系建议；
* SQLite 必须持久化用户创建的局、报名、局状态、群消息、反馈、信用、举报、关系空间和限制状态；Docker 部署使用持久卷；
* 统一 Agent 下的合作、玩伴、相亲、招聘、创投和旅友分别拥有独立关系空间；当前空间的会话、Intent 和公开资料不得进入其他空间的推荐；
* 信任体系必须包含邮箱/手机/工作学校/主理人/可选实名/机构背书、信用分、完成局/爽约/成立举报记录；
* 用户可举报并提交证据链接、纠正数据、请求删除和申诉；管理员可核查举报、调整认证、限制功能、临时封禁、永久封禁和解除限制；
* 简单管理后台：数据概览、模型平台、活动与报名、局加入规则、生命周期、群入口、信任治理、认证与封禁、内测码、用户列表、Agent 日志。

## 2.2 当前版本不做

* 用户之间私聊；
* 好友或关注关系；
* 图片、语音、视频消息；
* 在线支付；
* 微信或手机号登录；
* App、小程序；
* 自动交换联系方式；
* 复杂推荐模型或向量召回；
* 多 Agent 工作流；
* 自动判断身份证件真伪或接入第三方实名服务；当前认证由管理员人工标记；
* 自动外呼、短信、微信模板消息或日历推送；当前提醒在局详情内展示；
* 管理员在后台编辑活动全部富文本内容；当前管理加入规则、生命周期、报名、群入口和治理；
* 真实邮箱验证码服务。

当前 Demo 优先完成：

```text
能自然聊
→ 能理解和修正
→ 能确认意图
→ Agent 能搜索并解释真实候选
→ 能推荐或创建局
→ 能按规则审核或加入
→ 能在局内协调、行动、反馈
→ 能积累信誉并处理风险
```

活动现场并行闭环：

```text
导入签到表
→ 浏览器剔除联系方式
→ 规则筛候选并优先覆盖每位嘉宾
→ Agent 选择最高价值关系并给出理由
→ 可视化对接图与现场引荐队列
→ 主办方标记已完成引荐
```

# 三、专业 Channel 策略

## 3.1 产品原则

攒攒采用“入口统一、能力专业化”的结构。

底层统一：

* 账号、基础 Profile 与会话；
* Intent 和 Opportunity 数据结构；
* Agent 工具、模型平台和数据库；
* 搜索、匹配、报名、邀请与管理后台；
* 现场攒攒等活动运营能力。

每个专业能力在内部继续独立配置：

* Agent 身份、系统提示词与访谈问题；
* Channel 内有效用户池与机会池；
* 匹配规则、结果卡片和状态文案；
* 隐私要求、内测码来源和未来收费产品。

用户侧不感知“进入另一个 Channel”，但系统必须记录当前会话选择的能力；第一期不进行跨能力自动匹配。旧数据中的 `startup` 和 `love` 仅作为内部兼容字段，新逻辑以 `channel` 为主要隔离键。

## 3.2 攒攒合作伙伴

定位：

> **不急着合伙，先找到值得聊和试合作的人。**

覆盖找合伙人、早期核心成员、值得加入的项目、短期合作伙伴和两周试合作。Agent 重点理解项目方向与阶段、当前团队、角色缺口、双方能力与资源、全职或兼职投入、薪资或股权、风险边界与试合作意愿。

用户侧结果使用：深聊、合作机会、共创局、试合作、合伙人候选。第一阶段成功定义为完成一次深度交流、进入一次真实试合作或双方愿意继续讨论，不承诺直接找到合伙人。

这是第一优先 Channel，利用现有 AI 创业社群与活动资源完成冷启动。

## 3.3 攒攒玩伴

定位：

> **想去做什么，告诉攒攒，它帮你找到合适的人一起去。**

第一期只支持近期、低风险、边界清楚的活动，如羽毛球、看展、咖啡、饭局、桌游、City Walk 和公开活动。暂不支持高风险户外、拼车、过夜旅行、大额消费、未成年人活动和私密邀约。

Agent 优先理解活动、时间、地点、人数、水平或偏好、群体氛围、排除项和发起意愿。结果使用活动局、搭子和小组，不把对话扩展成长画像问卷。该 Channel 用于验证高频使用、留存与复购。

## 3.4 攒攒相亲

定位：

> **先匹配关系意图，再匹配人。**

Agent 重点理解关系目标、认识方式、城市与居住计划、婚育计划、生活节奏、见面频率、必须条件、排除条件，以及用户能为关系提供什么。候选只来自当前相亲用户池或对应合作方人群池。

用户侧结果使用一对一认识机会、小型认识局、双方匿名匹配和阶段性认识计划；状态使用信息了解中、匹配中、等待双方确认、双方愿意认识、已完成介绍和等待反馈。第一期通过红娘机构、校友会或青年社群联合测试，不独立大规模获客。

## 3.5 攒攒招聘

定位：

> **不只是匹配简历，而是匹配双方真正想要的工作机会。**

进入后先区分“我在招人”与“我在找工作”。招聘方访谈岗位真正解决的问题、必须能力、团队阶段、薪资地点与入职时间；求职者访谈目标方向、核心能力、公司阶段偏好、薪资地点、风险偏好、入职时间与排除项。

用户侧不使用“局”，统一使用岗位机会、人才候选、双向沟通、面试邀请和职业交流。该 Channel 待创业搭档场景出现稳定招人和求职需求后再正式拆分运营。

## 3.6 攒攒创投

定位：

> **让好项目和合适的资金先聊起来。**

进入后先区分“我在找资金”“我在找项目”与“我在提供创投服务”。项目方重点说明行业、阶段、产品与数据进展、团队、融资轮次、计划金额、资金用途和时间表；投资方重点说明关注行业、投资阶段、单笔范围、地域、决策流程和排除项。

用户侧统一使用创投机会、项目方、投资方、交换材料和后续会谈，不使用“系统评分”或“保证融资”等表达。攒攒只促成一次有准备的双向交流，不提供投资建议，不承诺融资成功或投资回报，也不替任何一方完成尽调和投资决策。

## 3.7 攒攒旅友

定位：

> **先找行程和边界合得来的人，再一起出发。**

覆盖周末短途、三至五天旅行、一周左右长线以及已有路线寻找同行者。Agent 重点理解目的地、出发城市、日期与时长、预算、交通住宿、旅行节奏、兴趣偏好、相关经验、安全边界和责任分工；信息不完整时优先用不超过两题的轻表单确认计划成熟度与旅行时长。

用户侧统一使用旅友、旅行计划、旅行同行和出发前见面。攒攒不代订交通与住宿、不代收旅费、不要求护照或身份证信息，也不承诺同行者可靠或旅途安全；默认建议双方在公共场所先见面，分别确认后再自行预订。第一期不支持未成年人、高风险户外、陌生人拼房、代购票和平台担保。

## 3.8 入口、链接与投放

当前网页 Demo 提供：

```text
/         攒攒产品 Landing Page
/chat     统一对话与全部功能
/admin    管理员后台
```

旧场景链接统一重定向到 `/chat`。场景方投放可通过 `mode` 与 `partner` 参数预选功能并记录来源，但不能因此生成一套独立页面、账号或导航。内测码下一阶段增加能力授权与 `partner` 归属，用于确定可用功能、候选池、后台统计和合作方分成。

## 3.9 活动现场对接

面向已经收集签到表的活动主办方，覆盖：

* 嘉宾需求与可提供资源的批量梳理；
* 需求方到资源方的定向连接；
* 双方均能回应彼此需求的双向互补；
* 优先覆盖每位存在候选关系的嘉宾，减少现场孤立；
* 给出可执行的引荐理由和现场第一句话；
* 由主办方人工确认并标记引荐，不自动交换联系方式。

该场景是跨 Channel 共用的活动运营工具，不改变普通用户的长期 Profile 或当前 Intent，也不自动把签到表嘉宾加入任一长期候选池。

# 四、核心对象

## 4.1 User：用户

保存：

* 邮箱；
* 用户昵称；
* 注册和账号状态；
* 是否允许继续匹配；
* 使用的内测码；
* Agent 理解进度；
* 角色：普通用户或管理员；
* 创建时间。

用户资料页不强调永久“所属 Channel”，但服务端应保存用户获得授权的 Channel 列表、来源合作方和当前活跃 Channel。

## 4.2 Profile：长期画像

表示相对稳定的信息：

* 所在城市；
* 当前身份；
* 能力或兴趣；
* 可以提供的价值；
* 一句话简介；
* 更新时间。

Profile 回答：

> 这个人是谁，以及通常能为一段关系带来什么。

## 4.3 Intent：当前意图

保存：

* 用户原始表达；
* Agent 生成的标题和摘要；
* 内部场景标签；
* 目标对象；
* 目标结果；
* 必须条件；
* 偏好条件；
* 排除条件；
* 用户可以提供的价值；
* 有效期；
* 状态。

Intent 回答：

> 这个人此刻想认识谁、为什么，以及什么样的匹配才有意义。

同一时间只保留一个最新的可操作 Intent。生成修订版时，旧版关闭并保留历史。

## 4.4 Opportunity：局

保存：

* 局名称；
* 内部场景标签；
* 局类型；
* 描述和群体摘要；
* 发起用户；
* 最低和最高人数；
* 当前人数；
* 活动开始和结束时间；
* 报名截止与取消截止时间；
* 城市、场地、详细地址和可选距离；
* 免费、AA 或固定价格及费用说明；
* 发起人姓名、角色和核验状态；
* 即时确认或发起人审核的报名方式；
* 当天流程和参加须知；
* 参与条件；
* 标签；
* 当前状态；
* 截止时间；
* 创建和更新时间。

一对一也是一种局：

```text
最低人数：2
最高人数：2
```

## 4.5 EventRegistration：活动报名关系

服务端保存用户和活动之间的报名关系：

* `pending`：等待发起人确认；
* `confirmed`：报名已确认，可查看报名后群入口；
* `waitlisted`：活动已满或由管理员转入候补；
* 用户报名说明；
* 报名时间和更新时间；
* 同一用户对同一活动最多一条有效记录。

取消报名会删除本条记录；超过活动的取消截止时间后需联系活动方处理。

## 4.6 EventJoinChannel：报名后沟通入口

保存：

* 活动 ID；
* 沟通方式：企业微信、微信或暂不提供外部群；
* 入口名称；
* 群入口 / 活动方承接页 URL；
* 加入说明；
* 是否启用。

只有 `confirmed` 用户可从用户 API 获得该入口。待确认、候补和未报名用户即使知道活动 ID，也不得获得入口链接。

## 4.7 RelationshipSpace：关系空间

统一账号下为 `founder / play / love / jobs / capital / travel` 分别保存：

* 当前空间的身份、技能、可提供价值和简介；
* 当前空间的对话、Intent 与推荐结果；
* 是否允许在该空间被发现；
* 最后更新时间。

昵称、头像、邮箱、城市与受控联系方式属于账号基础资料；身份表达、能力、提供价值、私密意图和推荐上下文属于关系空间。推荐 API 必须按 `channel` 读取对应空间资料，禁止自动拼接其他空间内容。

## 4.8 RoomState：局的行动状态

保存状态、约定时间、地点、会议链接、小目标、角色分工、截止日期、完成标准、是否继续和更新时间。状态采用完整行动生命周期，发起人和管理员可推进；报名人数达到最低成局人数时，早期状态可自动进入“已成立”。

## 4.9 RoomMessage / RoomFeedback：局内协作结果

`RoomMessage` 只向发起人和已确认成员开放，保存作者、正文与发送时间；不把消息正文交给推荐模型。`RoomFeedback` 保存是否到场、完成情况、继续意愿、评分和说明，并增量更新完成局、爽约与信用分，编辑反馈不得重复累计。

## 4.10 TrustProfile / TrustReport / UserRestriction：信任与治理

`TrustProfile` 保存认证、信用分、完成局、爽约和成立举报数；`TrustReport` 保存提交人、涉及成员、相关局、类别、说明、证据链接和处理状态；`UserRestriction` 保存正常、功能受限、临时封禁或永久封禁及原因和期限。举报只有管理员判定“已处理/成立”后才影响被举报用户的信用数据，申诉可在限制期间继续提交。

## 4.11 EventGuest / EventMatch：活动临时对象

`EventGuest` 由 Excel 在浏览器内临时解析，包含完整姓名、角色、项目阶段、职务、核心需求和可提供资源；微信、手机号等联系方式不进入该对象。服务端只接收匿名 ID 及完成匹配必需的非联系字段。

`EventMatch` 保存本轮可解释关系：需求方、资源方、强度、匹配维度、理由、现场开场建议、是否双向互补。当前 Demo 中二者只存在页面内存，不写入 LocalStorage 或仓库，刷新页面即清空。

# 七、详细页面设计

## P01 注册与登录页

### 页面目标

用最短路径完成邀请制注册或登录，不要求用户在登录前理解功能体系或决定场景。

### 页面结构

* 页面采用单列居中结构，不使用左右分栏；
* 左上显示攒攒品牌，中央依次显示品牌图形、“登录或加入”、一句说明、账号方式与表单；
* 背景使用极浅点阵纹理，不放大标题、不展示场景插画和产品功能列表；
* “加入内测”和“已有账号”在同一位置切换；
* 未接入真实 OAuth 前，不展示 Google、微信等不可用的第三方登录按钮。

### 注册字段

注册分两步完成，避免首次进入同时面对四个字段：

1. 邮箱、邮箱验证码；
2. 昵称、内测码。

第二步允许返回修改邮箱，已填写的验证码和内测码不丢失。

### 登录字段

* 邮箱；
* 邮箱验证码。

### Demo 规则

* 默认体验内测码：`CUANCUAN2026`；
* Demo 验证码：`888888`；
* 正式版本必须接入真实验证码服务。

### 不再出现

* 创业/恋爱 Channel 二选一；
* 机器人或星芒 AI 图标；
* 与产品无关的功能宣传。

## P02 攒攒对话页

### 页面目标

让用户像聊天一样表达，而不是完成问卷。

### 页面结构

```text
浅灰侧栏：品牌 → 新对话 → 工作台导航 → 账号
→ 工作区左上“攒攒 1.0”Agent 选择器
→ 独立滚动的对话区
→ Intent / 推荐 / 新局卡片
→ 固定在布局底部的输入区；当前能力位于输入框左下，发送为右下圆形图标按钮
```

### 对话原则

1. 首问开放，不预设用户属于哪个场景；
2. 用户可以一次说很多，也可以只说一句；
3. Agent 必须结合全部历史消息理解；
4. 每次最多追问一个最关键的信息缺口；
5. 用户一次说够时直接整理 Intent；
6. 用户可以随时补充、纠正、换方向；
7. Agent 不重复已理解的问题；
8. Agent 不自动替用户加入局。

### 核心理解维度

* Goal：想找到什么人或机会；
* Context：城市、阶段、生活或项目处境；
* Offer：用户可以提供什么；
* Commitment：愿意怎样投入或开始；
* Constraints：边界、必须项和排除项。

理解到至少四个维度时，可以生成 Intent 草稿。若对话较长，允许在至少三个维度明确时先生成可修订版本。

### Intent 确认卡

展示：

* Intent 标题；
* 对整段对话的摘要；
* 用户想找到什么；
* 用户可以提供什么；
* 有效期；
* “继续聊聊修改”；
* “方向没问题，开始匹配”。

只有用户确认后，Intent 才进入 `active`。

### 新对话

用户可从对话页顶部直接开始新对话；若当前没有有效对话内容，则直接聚焦输入区，不重复确认。已有内容时经用户确认后：

* 清空当前对话消息；
* 删除当前自动画像；
* 关闭当前草稿、匹配中或暂停的 Intent；
* 理解进度归零；
* 保留已经加入或创建的局。

## P03 Agent 匹配结果卡

展示最多四个优先推荐。生成推荐前，Agent 必须在当前关系空间先调用搜索工具，再查看候选局详情。每张卡展示：

* 局类型；
* 局名称；
* 当前人数；
* 群体摘要；
* 匹配判断与可解释分数；
* 引用用户目标、可提供价值和候选成员事实的理由；
* 能力互补点、加入前待确认边界与建议下一步；
* 查看详情；
* 加入或退出；
* 没有合适结果时创建新局。

推荐卡只展示最新一组结果，避免历史卡片重复堆叠。模型不可用时可使用本地可解释打分，但必须标明来源，不得伪装成模型判断。

## P04 发现新局

### 展示范围

默认展示当前关系空间内未结束的公开机会，用户可主动选择“全部”或其他场景查看公开局。Agent 推荐、个性化解释和空间资料始终只使用当前关系空间；`invite_only` 局只向发起人和已有报名关系的用户返回。合作伙伴、玩伴、相亲、招聘、创投和旅友的列表语言与结果类型不同，不允许在后台自动跨空间匹配。

### 筛选

* 全部与从今天开始的七日日期条；
* 当前 Channel 内的细分类；
* 多人局；
* 即将满员。
* 城市；
* 为你推荐、按时间、按距离排序。

### 局卡字段

* 局类型和场景标签；
* 月份、日期和星期；
* 名称；
* 群体摘要；
* 开始 / 结束时间；
* 场地、距离和价格；
* 当前人数和剩余名额；
* 报名状态；
* 查看详情；
* 申请加入、即时确认或加入候补。

## P05 局详情

详情页只服务三个连续判断：这是一个什么局、时间地点是否合适、现在是否参加。首屏按以下顺序展示：

* 局名称、类型和一句话摘要；
* 日期、开始 / 结束时间、场地、详细地址和距离；
* 费用、当前人数、最低成局人数与容量；
* 发起人姓名和核验状态；
* 当前报名状态或唯一主报名动作。

继续向下展示：

* 活动目的和详细说明；
* 攒攒的群体观察，合并在活动说明之后，不单独占用侧栏模块；
* 当天流程；
* 参加者全名、摘要、角色、需求和可提供价值；
* 参加须知，合并在报名区之后；
* 可选的报名说明输入、报名截止和取消截止；
* 待确认、已确认、候补状态；
* 发起人可见申请列表、申请说明与确认/候补动作；
* 报名确认后开放站内局内群聊，并按活动方配置展示微信 / 企业微信群入口；
* 取消报名。

详情页在报名判断后继续展示行动进度、攒攒当前下一步、时间提醒、小目标与分工、会后反馈和下一步关系建议。发起人可在同一详情中修改加入规则与行动安排，不依赖管理员代操作。

### 信息层级与视觉约束

* 不使用中英双语眉题、独立编号、重复状态解释和装饰性脚注；
* 不把发起人、推荐原因、报名、须知分别做成多个竞争卡片；
* 桌面端报名区可在内容滚动时保持可见，移动端报名区排在详细说明之前；
* 报名卡是详情页唯一需要卡片表面的交互区域，其余内容依靠留白和细分隔线组织；
* 正文和辅助信息保持可直接阅读的字号，不以大量小字压缩信息；
* 已报名用户只看到当前状态、取消入口和有权限访问的群入口，不再重复展示申请表单。

### 报名后沟通与行动闭环

攒攒内的局内群聊是已确认成员的默认协调空间，负责时间、地点、会议链接、分工与下一步；消息保存在 SQLite，只向发起人和 `confirmed` 成员返回，不发送给推荐模型。微信 / 企业微信群作为可选外部入口，继续承接集合和临时沟通，但不能替代攒攒中的报名状态与行动事实。

群入口默认不公开：管理员配置后，也仅通过鉴权后的活动 API 返回给 `confirmed` 用户。若暂未配置链接，详情页明确显示“活动群入口将在活动前开放”。

局的早期状态由报名数据辅助推进：出现待审核申请时进入“待确认”，确认人数达到最低人数时进入“已成立”；空位释放时按报名时间自动补最早候补。已预约及以后的状态由发起人或管理员明确更新，避免系统凭时间猜测真实行动结果。

### 展示与匹配原则

* 未导入时使用 GPT Image 2 生成的三张独立无框卡通线稿依次呈现“创业局 / 婚恋局 / 兴趣局”，不能只替换标题而复用同一张人物构图。人物使用圆润抽象造型、少量黑色线条、几乎无五官的成熟产品插画语言，不使用真人速写；婚恋图需通过相向坐姿、共享甜点和单枝小花建立清晰但克制的约会氛围；

* 姓名只用于主办方当前浏览器中的可视化，不发送给模型；
* 每位存在真实候选关系的嘉宾应尽可能至少拥有一条连线；

# 八、Agent 核心能力

## 8.1 开放式对话理解

Agent 必须：

* 分析全部用户消息，不只看最后一句；
* 将长表达拆分为多个有效线索；
* 动态识别场景；
* 增量更新 Profile；
* 维护理解完整度；
* 只追问最关键的缺失信息；
* 在信息足够时生成 Intent 草稿。

当前 Demo 使用 Vercel AI SDK `ToolLoopAgent` 编排对话和工具调用，底层模型为阶跃星辰开放平台。模型只负责结合对话上下文生成自然回复、理解进度、结构化 Intent 草稿，以及在合适时生成 `ask_user_question` 表单参数；Intent 确认、加入、退出和创建局等业务动作仍由产品代码校验和执行，模型不能直接替用户改变业务状态。

### 8.1.1 轻量提问表单

Agent 可以在用户表达很短、方向模糊，或用户明确希望通过选项回答时调用 `ask_user_question`。使用规则：

* 每轮最多调用一次，每张表单最多 3 个问题；
* 支持单选、多选和一句话简答，选项一般包含“还不确定 / 都可以”或允许“其他”；
* 优先询问关系目标、开始方式、投入方式、城市范围和关键边界；
* 不得把固定五题问卷重新包装成表单，也不得要求用户选择内部场景标签；
* 信息已经足够生成 Intent 时不得发表单；
* 用户提交表单后的下一轮不连续调用表单，优先吸收答案生成可修订草稿，或用一句自然语言补问剩余关键缺口；
* 表单答案必须使用用户看到的标签回传上下文，不得展示 `side_project` 等内部值。

## 8.2 场景识别

内部兼容字段仍保留：

* `startup`：合作、合伙、招聘、加入团队、项目、客户、技术、销售等；
* `love`：恋爱、长期关系、伴侣、约会、认真认识、交友、兴趣活动等。

Channel 已由入口确定；Agent 不再从一次对话中猜测应进入哪个产品。`startup` 与 `love` 只服务旧数据兼容，用户不接触技术标签。

## 8.3 搜索已有局

先取当前 Channel 内所有未结束机会，再按当前 Intent 排序，同时考虑：

* 是否已加入；
* 是否已满员；
* 目标相容性；
* 用户能否补足角色；
* 参与方式是否匹配。

## 8.4 创建局

没有合适结果时，Agent 可根据当前 Intent 创建：

* 创业搭档：深聊、共创、试合作或小型多人局；
* 去玩：活动局、搭子或小组；
* 相亲：一对一认识或小型认识局；
* 招聘：岗位机会、人才候选或双向沟通。

创建后，发起用户自动成为 host 和第一位成员。

## 8.5 推荐解释

解释必须包含：

* 为什么目标相容；
* 用户可能获得什么；
* 用户能提供什么；
* 当前仍有哪些不确定性；
* 不替用户做加入决定。

## 8.6 活动批量匹配

活动匹配采用“规则候选层 + Agent 判断层 + 覆盖率补齐层”：

1. 规则候选层只在技术、行业场景、客户渠道、数据、资金、产品共创、增长内容、行业认知或明确共同主题存在证据时建立候选；
2. Agent 只接收匿名候选及必要的需求/资源摘要，从最高质量候选中选出优先引荐关系，不允许改写资源事实；
3. 服务端只用当前这一对嘉宾的原始需求与资源确定性生成理由和开场建议，避免跨候选串信息；
4. 覆盖率补齐层优先为尚未出现的嘉宾选择其最强真实候选，再用剩余名额补充高价值关系；
5. 模型失败时直接使用同一候选和覆盖率逻辑生成关系，保证现场不中断；
6. 模型不得改写候选方向、虚构匹配维度或把“强度”解释为概率。

# 十、简单匹配机制

## 10.1 基础过滤

* 局未结束；
* 仍有名额；
* 用户未加入；
* 当前状态允许加入。

## 10.2 排序

优先级：

1. 与当前 Intent 内部场景一致；
2. 目标和局类型相容；
3. 用户可以补足当前角色；
4. 城市、投入方式和节奏相容；
5. 当前人数接近成局。

当前 Demo 使用规则优先，不宣称复杂算法能力。

## 10.3 活动关系覆盖

活动候选先按匹配强度、双向互补和维度数量排序，再执行覆盖率选择：每轮优先选择能带入最多未连接嘉宾的真实候选；达到当前可覆盖人数后，用更高质量候选填满引荐队列。关系上限随人数增长，当前最多 24 组，避免 40 人场次图面失控。

## 13.3 后端

当前 Demo：

* `/api/chat` Route Handler 处理 Agent 对话理解；
* `/api/event-match` Route Handler 处理匿名嘉宾校验、规则候选生成、覆盖率补齐和活动 Agent 判断；
* `GET /api/events` 返回当前关系空间可见的公开局、当前用户报名状态和发起人身份；邀请制局不向无关用户泄露；
* `POST /api/events/:eventId/registration` 在服务端提交或取消报名，并校验报名与取消截止时间；
* `GET /api/recommendations` 使用 `ToolLoopAgent` 的 `search_rooms`、`inspect_room` 工具读取真实局数据，结合当前空间画像生成个性化推荐；异常时使用相同数据做可解释规则降级；
* `GET/POST /api/rooms/:eventId` 返回和更新局工作区、审核申请、生命周期、协作计划、群聊、反馈与举报；所有写操作由服务端校验发起人或成员权限；
* `GET/POST /api/admin/events` 返回活动运营数据、调整报名状态、加入规则、发现范围、生命周期并配置群入口；
* 服务端使用 Vercel AI SDK `ToolLoopAgent` 编排模型与工具，使用 `@ai-sdk/openai-compatible` 连接阶跃星辰；
* 服务端读取模型环境变量并代理请求，客户端只获得整理后的回复、进度、Intent 草稿或经过 Schema 校验的表单参数；
* 模型输出必须经过 JSON 提取、字段校验、长度限制和场景枚举校验；
* `ask_user_question` 使用 Zod Schema 限制标题、问题数量、问题类型和选项数量；工具没有服务端业务执行函数，只返回待用户操作的表单；
* 单次模型请求超时为 15 秒；对超时、限流、5xx、网络错误和异常 JSON 自动重试一次，仍失败时使用本地规则降级；
* 活动匹配只调用一次低推理 Agent，单次上限 25 秒；失败时立即返回覆盖率优先的规则候选，不做第二轮长等待；
* 模型请求期间允许切换页面，请求在后台继续；结果只更新聊天状态，不在“发现新局”等其他页面弹出模型提示；
* 只有用户主动“重新聊”或退出登录时，才取消尚未完成的聊天请求；
* 确认 Intent 仍由确定性前端状态执行；创建局、报名、审核、候补补位、生命周期、群聊、反馈和治理均通过鉴权后的服务端接口写入 SQLite，不接受模型绕过权限直接写入。

生产化目标：

* 单一 `/api/app` 路由处理业务动作；
* HttpOnly Cookie 会话；
* D1 持久化；
* 每次动作后返回最新 Bootstrap 数据；
* Agent 规则和所有写操作在服务端执行。

# 十五、核心状态

## 15.1 Intent

```text
draft → active → paused → active
draft / active / paused → closed
```

## 15.2 Room

```text
recruiting
→ pending_confirmation
→ formed
→ scheduled
→ in_progress
→ completed / cancelled
→ follow_up
```

有待审核申请时自动进入 `pending_confirmation`；确认人数达到最低成局人数时自动进入 `formed`。之后由发起人维护预约、进行、完成 / 取消和后续连接状态，并同步时间、地点、会议链接、目标、角色、截止日期和完成标准。

## 15.3 EventRegistration

```text
无报名 → pending / confirmed / waitlisted
pending ↔ confirmed ↔ waitlisted
pending / confirmed / waitlisted → 取消报名
```

审核型活动初次报名进入 `pending`；即时确认型活动在有名额时进入 `confirmed`；满员时进入 `waitlisted`。

## 15.4 Invite Code

```text
active ↔ disabled
```

# 十七、隐私与安全原则

1. 对外只展示生成后的成员摘要，不展示原始对话；
2. 邮箱和敏感联系方式不展示给其他用户；
3. 用户加入前必须主动点击确认；
4. 新对话只清理当前对话和意图，不误删已加入的局；
5. 所有修改和删除操作限定当前用户数据；
6. 内测码校验使用原子更新；
7. 会话 Cookie 使用 HttpOnly 和 SameSite；
8. 正式版本必须增加频率限制、日志脱敏和内容安全策略。
9. 模型 API Key 只允许存在于服务端环境变量或 `model_providers` 加密字段，不进入客户端包、普通业务数据或 Agent 日志；浏览器只显示掩码。
10. 用户对话会被发送给第三方模型服务用于当次理解，正式上线前必须在隐私说明中明确数据处理范围。
11. 活动页向当前主办方展示签到表中的完整姓名，但姓名、公司名、微信、手机号、邮箱和原始 Excel 不得进入活动匹配模型请求。
12. 活动签到表和关系结果当前只存在页面内存；公开仓库只允许保存合成演示数据，不得提交真实名单。
13. 活动群 URL 属于报名后权限数据，只能由服务端根据已确认报名状态返回；不能放入公开活动种子、HTML、客户端环境变量或未鉴权接口。
14. 微信 / 企业微信群只承担即时沟通，报名状态、时间地点和活动通知仍以攒攒数据库为准，避免外部群成为不可审计的唯一事实来源。

# 十九、MVP 验收标准

## 注册

* 根路径 `/` 展示产品 Landing Page，六组插画清楚解释产品能做什么；
* Landing 的主动作和场景说明最终都汇入 `/chat`，不出现多个独立产品入口；
* 旧 `/founder`、`/play`、`/love`、`/jobs`、`/capital`、`/travel` 链接自动进入 `/chat`；
* 合法内测码和验证码可以完成注册；
* 重复邮箱、无效码、用尽码、停用码和过期码有明确错误提示；
* 登录页使用统一产品文案，不要求用户提前选择需求类型。

## Agent

* 首问为开放式问题；
* 不出现固定五题问卷；
* 一条长消息包含足够信息时可直接生成 Intent；
* 信息不足时自然语言每次只追问一个关键问题；适合结构化选择时可展示不超过 3 题的轻量表单；
* 同一输入框下可选择合作、玩伴、相亲、招聘、创投、旅友和活动名单配对；
* 六类对话分别使用创业合作、近期活动、关系匹配、双向职业、双向创投和旅行同行提示词；
* 当前选择的专业 Agent 不主动搜索或推荐其他类型的候选；
* Agent 使用 `search_rooms` 和 `inspect_room` 读取当前关系空间的真实局数据，不以场景标签过滤冒充智能匹配；
* 推荐同时考虑目标与成员缺口、城市时间、投入方式、硬性边界和多人组合，并显示来自用户资料与局数据的个性化理由、约束和数据来源；
* 能生成 Profile 和 Intent；
* 配置 `STEP_API_KEY` 后，对话响应来自阶跃星辰模型，客户端无法读取密钥；
* Agent 可在合适时调用 `ask_user_question`，网页正确渲染单选、多选、简答和“其他”；
* 表单必填校验、提交、只读回显和中文答案续聊正常，且不会在提交后连续发表单；
* 未配置密钥或有限重试后模型仍异常时，聊天输入区明确显示“模型繁忙 · 本地完成”，主流程仍可继续；
* 模型请求过程中切换到“发现新局”等页面不会弹出跨页面提示，返回聊天后消息和 Intent 结果仍然保留；
* 用户可继续补充并生成修订版；
* 用户确认后 Intent 进入 active；
* 顶部“新对话”动作可结束当前对话与当前意图，并保留已加入或创建的局。

## 局

* 发现页统一展示开放的局，并可按合作、玩伴、相亲、招聘、创投和旅友筛选；默认当前关系空间，用户可主动切换到全部公开局；
* 未来七日、城市、场景、多人局和即将满员筛选可用，推荐 / 时间 / 距离排序可用；
* 列表清楚显示日期、时间、地点、距离、价格、参加人数和容量；
* 详情可查看完整时间地点、发起人、参加者全名、流程和须知；
* 报名、取消、待确认、已确认和候补状态可用；
* 满员时进入候补，不能绕过容量直接确认；
* 每个局可由发起人独立配置“需要审核 / 即时加入”和“公开 / 邀请制”；邀请制局不向无关用户返回；
* 发起人可在局工作区确认或候补申请，取消后最早候补自动补位；
* 报名和取消截止时间由服务端校验；
* 仅发起人和已确认成员可进入站内局内群聊；外部群入口只向已确认用户开放，未配置时有明确说明；
* 局支持招募中、待确认、已成立、已预约、进行中、已完成 / 已取消和后续连接完整生命周期；
* 发起人可维护时间、地点 / 会议链接、提醒、目标、角色、截止日期、完成标准和是否继续；成员可查看会前摘要、提交会后反馈并获得下一步关系建议；
* Agent 可创建新局；
* 用户创建或加入的局出现在“我的局”。

## 关系空间与信任治理

* 合作、玩伴、相亲、招聘、创投和旅友共用一个 Agent 入口，但身份、能力、资源、简介、对话和 Intent 按空间隔离；
* 切换空间后不会把相亲表达带入工作推荐，也不会把工作资料自动暴露给兴趣或恋爱场景；
* 资料页展示邮箱、手机、工作 / 学校、主理人、实名和机构背书状态，以及到场、正常取消、No-show、履约、继续意愿、举报数和信用分；
* 管理后台可人工维护认证、处理带证据的举报与申诉，并设置风险限制、临时封禁和永久封禁；
* 原始对话不公开，联系方式只在用户明确同意后使用；管理员概览和 Agent 日志不读取局内群聊正文。

## 交互

* 左上品牌不打开场景菜单；对话框下六个功能均可点击，切换专业能力时不跳转页面；
* 右下账号入口只打开“我的资料”和“退出登录”，不重复提供“回到对话”和“重新开始对话”；
* 我的资料支持选择六个扁平卡通 avatar、上传头像、修改昵称、城市、身份、简介、兴趣能力、可提供资源和微信号，并在重新登录后回显；
* 对话消息不被顶部和输入框遮挡；
* 只有对话区滚动，不推动整个页面异常跳动；
* 桌面端和移动端主要流程可用；
* 页面不出现机器人脸、星芒或 emoji 产品图标。

## 现场攒攒

* 当前 17 列签到表可导入，真实样表中的 10 位有效嘉宾全部识别；
* 页面显示嘉宾完整姓名，联系方式列数明确提示已排除；
* 浏览器发往 `/api/event-match` 的请求体不包含姓名、公司、微信、手机号和邮箱；
* 存在真实候选关系时，每位嘉宾都尽可能至少连接一人；无法连接时不得编造理由；
* 阶跃星辰成功时显示“Agent · 阶跃星辰”，模型异常时显示“本地规则降级”且仍能出图；
* 全部 / 优先对接 / 双向互补筛选、节点和连线详情、开场建议、引荐标记均可用；
* 桌面端关系图不裁切，移动端可横向浏览且底部导航不遮挡操作。

## 后台

* 管理员可查看数据概览；
* 可配置多种 OpenAI-Compatible 模型平台并安全保存密钥；
* 可查看活动和报名者说明，并调整待确认、已确认和候补状态；
* 可配置局加入审核、发现范围和行动生命周期；
* 可设置微信 / 企业微信群入口，且入口只向已确认用户返回；
* 可创建、启用和停用内测码；
* 可查看用户、信用分、行动记录、举报数和各项认证，并维护认证与限制状态；
* 可处理举报、证据、申诉和治理结论；
* 可查看 Agent 日志；
* 普通用户不能进入管理后台。

# 二十、当前版本最终闭环

```text
用户从 Landing Page 进入统一 `/chat` 工作台
→ 在同一个输入框下直接表达，或选择合作、玩伴、相亲、招聘、创投、旅友、活动名单配对
→ 对应专业 Agent 自然对话，不填固定长问卷
→ Agent 动态理解当前能力所需的目标、处境、双方价值、投入和边界
→ 生成一版可修改 Intent
→ 用户确认 Intent
→ Agent 用只读工具搜索并检查当前关系空间中的真实局数据
→ 向用户解释匹配依据、成员缺口、时间地点、限制和数据来源
→ 用户按日期和城市查看局，读清参与者、加入规则与协作方式
→ 用户主动报名并获得待确认、已确认或候补状态，或让攒攒创建新局
→ 发起人审核，候补自动补位；确认成员进入站内局内群聊并按需查看外部群入口
→ 局沿招募、成立、预约、进行、完成和后续连接推进，记录协作计划、反馈和行动信誉
→ 用户在“我的局”中持续管理，后续对话继续校准当前关系空间的推荐
```

这一闭环是当前邀请制 Demo 的核心产品边界。

活动场景作为第二条 Demo 闭环：主办方导入签到表后，攒攒在不上传联系方式和姓名的前提下整理需求—资源关系，优先让每位有真实候选的嘉宾进入合作图，再由主办方现场完成和记录引荐。
