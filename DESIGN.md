# DESIGN.md — 攒攒

## 1. 设计主张

攒攒是帮助用户说清真实意图、连接合适的人与机会的关系 Agent。界面首先让用户愿意表达，其次帮助用户判断和行动，最后才展示系统能力。

视觉采用 **Calm Social Workspace**：

- 借鉴 Manus 公开界面的克制留白、任务优先和清晰快捷入口；
- 以浅灰工作台承载白色内容表面，减少无意义装饰；
- 以近黑色表达品牌、当前选择和主动作，用暖灰层级建立成熟的工具感；
- 智能通过上下文记忆、可解释推荐和低压力表单体现，不通过机器人、魔法棒或发光球体现；
- 创业、招聘、现场攒攒和关系交友共用一套视觉语言。

关键词：`克制`、`成熟`、`可信`、`任务清晰`、`真实连接`。

## 2. 体验原则

### 2.1 对话优先，表单减压

用户可以直接说，也可以点击 2–3 个开场建议。Agent 需要补充结构化信息时，可调用 `ask_user_question` 渲染轻量表单；表单应解释为什么要问，并允许用户少量点击完成。

### 2.2 一屏一个主动作

- 登录页：理解价值并进入；
- 对话页：继续表达；
- 发现页：判断是否值得靠近；
- 现场攒攒：生成关系图或执行现场引荐；
- 我的局：管理持续中的关系；
- 我的资料：校正长期信息和当前意图。

同一视口只保留一个视觉最强的主按钮。

### 2.3 稳定比炫技重要

页面切换不能依赖模型请求。模型繁忙时只在发起请求的局部显示降级状态，不用全局错误打断浏览。用户切换页面时，对话请求可以继续或安全取消，但不能弹出无关故障提示。

### 2.4 关系需要边界

不使用“缘分值”“灵魂契合度”等夸张表达，不把人做成可快速左右滑动的商品。推荐必须给出判断线索；活动关系图展示全名，但联系方式和原始回答不进入模型。

### 2.5 入口统一，能力专业

攒攒母品牌负责统一视觉、账号、Agent 交互和机会管理。用户始终进入同一个对话工作台，再通过输入框下方的轻量功能选择合作伙伴、玩伴、相亲、招聘、创投、旅友或活动名单配对；不再把同一产品拆成多个入口和登录页。

- 合作伙伴：克制、行动导向，强调深聊与试合作；
- 玩伴：轻松但不幼稚，强调近期活动、时间地点和氛围；
- 相亲：温和、可信，强调关系意图与双方确认，不使用粉色情感模板；
- 招聘：专业、直接，强调岗位问题与双向成立，不使用“局”作为主要名词；
- 创投：专业、双向，先区分项目方与投资方，再说明阶段、方向和下一步；不承诺融资结果或投资回报。
- 旅友：轻松但有安全边界，强调目的地、日期、预算、节奏、责任分工与出发前见面，不使用旅游预订平台的促销语言。

六类能力共用近黑、暖灰、留白和同一品牌图标；专业感来自 Agent 问法和结果语言，不通过换页面或更换高饱和主题色表达。

## 3. 视觉系统

### 3.1 用色比例

- 约 75% 为 `canvas`、`surface`；
- 约 20% 为文字、灰色层级和分隔线；
- 不超过 5% 为 `accent`；
- 近黑色用于品牌、当前选择和主动作；彩色只在必须表达业务状态时出现；
- 成功、等待、危险色只表达状态，不参与装饰。

### 3.2 排版

- 中文优先使用系统无衬线字体，保证加载稳定；
- 登录页使用居中账号入口，主标题桌面约 `28px`、移动端约 `26px`；登录是工具步骤，不承担 Landing 的大标题叙事；
- 工作台 H1 桌面 `38–54px`，对话页标题可降到 `22–25px`；
- 正文 `14–16px`，行高 `1.65–1.8`；
- 功能性辅助文字通常为 `12–13px`；首屏不使用 `9–11px` 的装饰性眉题、编号和脚注；
- 同一首屏最多保留一段辅助说明。删除后不影响理解的状态、标签和解释，应直接删除；
- 不使用编号代替导航图标。

### 3.3 表面、边框和阴影

- 默认卡片使用 `1px #E2E5EC` 边框；
- 主要内容卡片圆角 `18px`，弹窗 `24px`，小控件 `10–12px`；
- 阴影应自然、低对比，不使用硬偏移印刷阴影；
- 页面主体尽量依靠留白和分隔线建立层级，不把每一段内容都放进卡片。

## 4. 品牌与图标

- 品牌符号由 GPT Image 2 生成，为“两把椅子被拉到小桌边”的有机黑色线描；最终以透明 PNG 同时用于 favicon、侧栏品牌和 Agent 头像，不再使用几何 SVG 标识；
- 导航图标统一为约 `1.8px` 圆角线性图标；
- 不使用 emoji、机器人脸、星芒、魔法棒、闪电或混杂风格图标；
- 图标必须有文字或可访问名称，不让用户只靠猜测理解功能；
- 活动关系图中的节点和连线属于数据可视化，可使用更细的中性线条，选中时切换为近黑强调。

## 5. 核心组件

### 5.0 统一入口

- 登录页采用单列居中结构：品牌位于左上，中央依次为品牌图形、“登录或加入”、账号方式与当前步骤表单；背景只使用极浅点阵纹理，不使用左右分栏、大幅宣言或场景插画；
- 邀请注册分为两步：先填写邮箱和验证码，再填写昵称和内测码；已有账号只显示邮箱与验证码。没有接入真实 OAuth 前，不展示第三方登录假按钮；
- 根路径首先是产品 Landing Page，整体参考 Manus Download 的节奏：居中价值主张、大幅产品展示、六组能力说明、三步流程和最终 CTA；所有转化动作统一进入 `/chat`；
- Landing 主标题直接说“想找人一起做件事？交给攒攒。”，说明只回答用户需要提供什么、攒攒会做什么；不在用户页面使用“意图匹配、相容、机会生成”等内部术语；
- Landing 首屏下方保留“不同需求的人来到同一张桌前”插画，并放入大幅产品展示容器；六组能力各自使用合作、玩伴、相亲、招聘、创投、旅友场景图，图像仅辅助理解，不承载正文；
- Landing 顶部只保留品牌、“能做什么”“怎么开始”两个页内锚点和一个主按钮；首屏只允许一个标题、一句说明和一个主按钮；
- 六组场景使用“名称 + 一句用途 + 场景图”的说明结构，不显示进入箭头、编号、测试状态和冗余说明；
- 对话首屏只保留一个总问题、一个输入框和一行可换行的能力按钮；
- 能力按钮采用低对比圆角胶囊，名称必须是用户能直接理解的动作，如“找合作伙伴”“认真相亲”，不使用 Channel 等内部术语；
- 切换能力不跳转页面，只更新当前专业提示词、输入提示与后续 Agent 表单；
- “活动名单配对”与六类个人需求同处功能栏，点击后进入现有现场攒攒工具；
- 页面不混用不同能力的对话历史；Profile 与账号保持共用。

### 5.1 侧栏

- 桌面宽约 `244px`，浅灰背景；
- 品牌下方首先提供“新对话”，再显示工作台主导航；账号菜单不重复放置新对话动作；
- 当前项使用中性选中面，图标切换为近黑；
- 底部只保留账户入口，不用长期状态小字占据视线；
- 移动端改为五项底部导航，图标在上、文字在下。

### 5.2 对话与输入框

- Agent 消息默认不使用气泡，保持自然文本阅读；
- 用户消息使用浅暖灰气泡，最大宽度 74%；
- 首轮显示最多三个可点击开场建议，点击后填入输入框并自动聚焦；
- 工作台左上使用“攒攒 1.0”Agent 选择器，展开后显示各专业能力的名称和一句职责说明；
- 输入框固定在对话底部，白底、柔和边框和自然阴影；
- 当前能力以输入框左下的小型工具胶囊显示；功能快捷入口位于输入框下方，使用同一套 16px 线性图标；
- 聚焦时使用中性深灰边框与轻微抬升；
- 发送按钮为输入框右下的圆形图标按钮，只在有内容时可用。

### 5.3 Ask User Question

- 使用白色卡片、柔和边框和清晰问题层级；
- 单选项在桌面可两列，移动端一列；
- 选中态使用深灰边框或浅暖灰底；
- 提交后折叠为答案摘要，避免持续占用对话空间；
- 表单失败不清空用户已经选择的答案。

### 5.4 现场攒攒

- 导入区说明只读取角色、阶段、需求和资源；
- 空状态使用 GPT Image 2 生成的三张独立卡通线稿依次展示“创业局 / 婚恋局 / 兴趣局”：创业局呈现产品、技术、渠道的资源互补，婚恋局通过相向坐姿、共享甜点和单枝小花呈现克制的约会氛围，兴趣局呈现同城摄影漫步。人物使用圆润抽象造型、少量黑色线条和几乎无五官的 Manus 式产品插画语言，不使用真人速写；
- 场景图内不烧录姓名和中文关系文案，避免生成文字失真或绝对定位错位；每张图下方固定显示一行“需求 × 资源 × 成局理由”，由真实 DOM 文本渲染；
- 页面留白直接作为画布，去掉外层圆角框、底色、阴影、人物卡片、状态角标、编号、黑色标签块与中央结果卡，以品牌图标同源的松弛线描风格呈现，不使用旋转轨道或科技仪表盘；
- 三张图每张稳定停留约 10 秒，切换使用约 900ms 淡化；用户可点击底部三条指示线直接切换，也可在插画区域左右拖动 / 滑动，并可使用键盘左右方向键。手动切换后重新开始当前场景计时；移动端优先删除次要批注而不是继续压缩字号，减少动态效果模式关闭自动轮播但保留手动切换；
- 主办方视图显示完整姓名；姓名不发送给模型；
- 关系图优先让存在真实候选的每位嘉宾至少连接一次；
- 连线按“共同话题 / 值得聊 / 优先对接”分级，不展示伪精确百分比；
- 点击节点或连线后，在检查面板中解释需求、资源、引荐理由和建议开场白。

### 5.5 活动详情

- 视觉上是一张安静的活动邀请函，不是信息仪表盘；
- 首屏只承担标题识别、时间地点判断和报名决策，标题控制在 `32–46px`，不使用占据大半屏的海报式字号；
- 时间、地点、费用、名额形成一条四项事实带，避免拆成四张卡；
- 正文只保留“这个局做什么”“怎么进行”“谁会来”三个标题；
- 发起人与核验状态进入标题区，推荐理由并入活动说明，参加须知并入报名区；
- 报名区是唯一的强调表面，桌面端保持可见，移动端先于详细正文出现；
- 不使用中英双语眉题、装饰编号和重复隐私脚注，不以 `9–10px` 小字换取表面上的紧凑。

### 5.6 我的资料

- 资料页是一张安静的个人名片，不使用卡片拼贴；桌面端左侧以大头像为视觉锚点，右侧使用一列清晰表单；
- 提供六个由 GPT Image 2 生成的社交产品标准 avatar：扁平卡通头像、圆形柔和底色、清晰人物轮廓和克制表情，缩小到 `48px` 仍能区分人物；不使用肖像线稿、真人速写、照片或 3D 形象；选择时只使用细描边与轻微位移反馈；
- 支持上传 JPG、PNG、WebP，本地居中裁切后预览；
- 昵称、城市、身份、微信号、简介、兴趣与擅长、可提供资源进入同一保存动作；
- 微信号的隐私说明使用一行正常可读正文，不用多段小字；
- 账号菜单只保留“我的资料”和“退出登录”，对话切换由主导航和对话页顶部负责。

### 5.7 管理员后台

- 使用独立 `/admin` 工作台，桌面为左侧管理导航与右侧数据工作区，移动端改为底部五项导航；
- 概览数据采用分栏与横线组织，不使用仪表盘卡片拼贴；
- 模型平台、内测码、用户和日志使用高密度表格，状态通过文字、边框和小圆点同时表达；
- API Key 只显示掩码，编辑时留空代表保留；添加或编辑模型平台使用单一对话框；
- 默认模型切换、密钥迁入和停用内测码必须提供明确完成反馈，删除模型平台需要二次确认。

## 6. 动效与可访问性

- 页面、消息和浮层进入控制在 `160–300ms`；
- 悬停最多上移 `1–2px`；
- 输入聚焦、选中和加载状态必须同时有颜色之外的形状或文字反馈；
- 所有交互控件支持键盘和 `focus-visible`；
- 正文与背景对比度遵循 WCAG AA；
- 遵守 `prefers-reduced-motion`，关闭非必要动画。
