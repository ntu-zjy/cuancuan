# 一、产品定义

## 1.1 产品名称

# 攒攒

代码与仓库标识统一使用 `cuancuan`，含义来自“攒局”；不得写作 `zanzan`。中文产品名保持“攒攒”。

产品主张：

> **任何想认识的人，都可以说给攒攒听。**

核心介绍：

> 无论用户想找创业合作、招聘关键成员、加入团队，还是认真认识一个人，都从同一个对话入口开始。攒攒理解用户的真实意图，寻找合适的人或已有机会，并帮助用户攒成一个可以加入的局。

## 1.2 产品本质

攒攒不是资料卡浏览平台，不要求用户先选择频道，也不通过一套固定问卷完成匹配。

产品核心流程：

```text
用户自然表达
→ Agent 结合完整上下文理解目标、处境、价值、投入方式和边界
→ 生成一版可修改的当前意图
→ 用户确认意图
→ Agent 优先推荐相容的已有局
→ 没有合适机会时创建新局
→ 用户加入、退出和管理自己的局
→ 后续对话继续校准匹配
```

## 1.3 当前阶段验证目标

1. 用户是否愿意用自然对话表达真实关系需求；
2. Agent 是否能从非结构化表达中形成可用意图；
3. 不要求选择场景后，用户是否更容易开始对话；
4. 推荐理由是否足以支持用户判断是否加入；
5. 用户是否愿意确认意图、加入已有局或创建新局；
6. 创业合作、招聘组队、加入团队、恋爱交友等场景能否共用同一产品入口；
7. 哪类意图更容易产生有效匹配和真实参与。

# 二、当前版本边界

## 2.1 当前版本必须支持

* 邮箱注册和登录；
* 邀请内测码校验与限量管理；
* 统一匹配空间，不要求注册时选择 Channel；
* 用户与 Agent 进行开放式自然对话；
* Agent 结合历史上下文动态追问，每次只追问一个关键缺口；
* 当 1–3 个选择或短回答明显比开放式表达更省力时，Agent 可调用 `ask_user_question` 展示轻量表单；
* 一次输入包含足够信息时直接生成意图，不强制走固定轮数；
* 对话理解由服务端 Vercel AI SDK `ToolLoopAgent` 编排，并调用阶跃星辰文本模型；浏览器端不得接触模型密钥；
* 模型未配置、超时或上游异常时切换到可识别的本地演示逻辑，保证主流程可继续体验；
* 自动保存 Profile 和当前 Intent；
* Intent 可继续补充、纠正、重新整理；
* 用户确认 Intent 后开始匹配；
* 浏览工作合作、招聘组队、恋爱交友等全部开放局；
* Agent 优先推荐与当前 Intent 类型相容的局；
* Agent 创建新局；
* 用户加入和退出局；
* 查看局详情、成员摘要、人数和状态；
* 查看自己加入或创建的局；
* 编辑基础资料、暂停和恢复匹配；
* 场景启动菜单、账号菜单等可见入口必须有真实行为；
* 简单管理后台：数据概览、内测码、用户列表、Agent 日志。

## 2.2 当前版本不做

* 用户之间私聊；
* 局内群聊；
* 好友或关注关系；
* 图片、语音、视频消息；
* 在线支付；
* 微信或手机号登录；
* App、小程序；
* 自动交换联系方式；
* 复杂实名和风控系统；
* 复杂推荐模型或向量召回；
* 多 Agent 工作流；
* 管理员在后台直接编辑全部业务对象；
* 真实邮箱验证码服务。

当前 Demo 优先完成：

```text
能自然聊
→ 能理解和修正
→ 能确认意图
→ 能推荐或创建局
→ 能加入和管理
```

# 三、统一场景策略

## 3.1 产品原则

产品内不再将用户永久划分到“创业”或“恋爱”频道。

用户看到的是一个统一的攒攒：

* 注册时无需选择场景；
* 可以在同一入口表达不同类型的关系需求；
* Agent 从用户对话中自动判断当前 Intent 属于哪类场景；
* 发现页可以浏览全部开放局；
* 推荐时优先考虑与当前 Intent 相容的机会。

旧数据中的 `startup` 和 `love` 字段作为内部场景标签保留，用于兼容和排序，不再作为用户池隔离条件。

## 3.2 工作与共创

覆盖：

* 找创业合伙人；
* 找早期成员；
* 招聘关键岗位；
* 寻找值得加入的团队；
* 寻找项目、行业资源或客户；
* 发起短期试合作；
* 认识同方向创业者。

Agent 重点理解：

* 当前项目、团队或求职阶段；
* 真正需要的角色或机会；
* 用户能提供的能力和资源；
* 全职、兼职、远程、线下或试合作偏好；
* 时间投入；
* 必须条件、排除条件和风险边界。

## 3.3 关系与交友

覆盖：

* 认真恋爱；
* 长期关系或婚姻目标；
* 自然认识新的人；
* 扩大真实社交圈；
* 兴趣活动和轻量多人局。

Agent 重点理解：

* 用户希望进入怎样的关系；
* 城市、生活节奏和真实日常；
* 用户在关系中能带来什么；
* 希望如何开始认识；
* 价值观、相处方式和明确边界。

## 3.4 场景入口

左侧“你想认识什么人？”入口提供三种开场模板：

1. 创业合作；
2. 招聘与组队；
3. 恋爱与交友。

选择后只将一段可编辑的开场文字带入输入框，不自动发送，不替用户做决定。

# 四、核心对象

## 4.1 User：用户

保存：

* 邮箱；
* 用户昵称；
* 注册和账号状态；
* 是否允许继续匹配；
* 使用的内测码；
* Agent 理解进度；
* 角色：普通用户或管理员；
* 创建时间。

产品层不再展示“所属 Channel”。

## 4.2 Profile：长期画像

表示相对稳定的信息：

* 所在城市；
* 当前身份；
* 能力或兴趣；
* 可以提供的价值；
* 一句话简介；
* 更新时间。

Profile 回答：

> 这个人是谁，以及通常能为一段关系带来什么。

## 4.3 Intent：当前意图

保存：

* 用户原始表达；
* Agent 生成的标题和摘要；
* 内部场景标签；
* 目标对象；
* 目标结果；
* 必须条件；
* 偏好条件；
* 排除条件；
* 用户可以提供的价值；
* 有效期；
* 状态。

Intent 回答：

> 这个人此刻想认识谁、为什么，以及什么样的匹配才有意义。

同一时间只保留一个最新的可操作 Intent。生成修订版时，旧版关闭并保留历史。

## 4.4 Opportunity：局

保存：

* 局名称；
* 内部场景标签；
* 局类型；
* 描述和群体摘要；
* 发起用户；
* 最低和最高人数；
* 当前人数；
* 参与条件；
* 标签；
* 当前状态；
* 截止时间；
* 创建和更新时间。

一对一也是一种局：

```text
最低人数：2
最高人数：2
```

## 4.5 Participation：参与关系

保存用户和局之间的关系：

* host：发起人；
* participant：参与者；
* joined：已加入；
* exited：已退出；
* 加入时间和更新时间。

# 七、详细页面设计

## P01 注册与登录页

### 页面目标

用统一入口说明产品价值，并完成邀请制注册或登录。

### 营销信息

首屏明确告诉用户：

* 创业合作可以；
* 招聘和加入团队可以；
* 恋爱交友也可以；
* 不用先选择频道；
* 不用填写长问卷；
* 先说真实需求，再由攒攒理解和匹配。

### 注册字段

* 邮箱；
* 昵称；
* 内测码；
* 邮箱验证码。

### 登录字段

* 邮箱；
* 邮箱验证码。

### Demo 规则

* 默认体验内测码：`CUANCUAN2026`；
* Demo 验证码：`888888`；
* 正式版本必须接入真实验证码服务。

### 不再出现

* 创业/恋爱 Channel 二选一；
* 机器人或星芒 AI 图标；
* 与产品无关的功能宣传。

## P02 攒攒对话页

### 页面目标

让用户像聊天一样表达，而不是完成问卷。

### 页面结构

```text
当前状态和标题
→ 攒攒状态 + 重新聊
→ 独立滚动的对话区
→ Intent / 推荐 / 新局卡片
→ 固定在布局底部的输入区
```

### 对话原则

1. 首问开放，不预设用户属于哪个场景；
2. 用户可以一次说很多，也可以只说一句；
3. Agent 必须结合全部历史消息理解；
4. 每次最多追问一个最关键的信息缺口；
5. 用户一次说够时直接整理 Intent；
6. 用户可以随时补充、纠正、换方向；
7. Agent 不重复已理解的问题；
8. Agent 不自动替用户加入局。

### 核心理解维度

* Goal：想找到什么人或机会；
* Context：城市、阶段、生活或项目处境；
* Offer：用户可以提供什么；
* Commitment：愿意怎样投入或开始；
* Constraints：边界、必须项和排除项。

理解到至少四个维度时，可以生成 Intent 草稿。若对话较长，允许在至少三个维度明确时先生成可修订版本。

### Intent 确认卡

展示：

* Intent 标题；
* 对整段对话的摘要；
* 用户想找到什么；
* 用户可以提供什么；
* 有效期；
* “继续聊聊修改”；
* “方向没问题，开始匹配”。

只有用户确认后，Intent 才进入 `active`。

### 重新聊

用户确认后：

* 清空当前对话消息；
* 删除当前自动画像；
* 关闭当前草稿、匹配中或暂停的 Intent；
* 理解进度归零；
* 保留已经加入或创建的局。

## P03 Agent 匹配结果卡

展示最多两个优先推荐：

* 局类型；
* 局名称；
* 当前人数；
* 群体摘要；
* 为什么适合用户；
* 查看详情；
* 加入或退出；
* 没有合适结果时创建新局。

推荐卡只展示最新一组结果，避免历史卡片重复堆叠。

## P04 发现新局

### 展示范围

展示统一空间内全部未结束的开放局，不再按用户 Channel 隔离。

### 筛选

* 全部；
* 工作与共创；
* 关系与交友；
* 多人局；
* 即将满员。

### 局卡字段

* 局类型和场景标签；
* 名称；
* 群体摘要；
* 标签；
* 当前人数和剩余名额；
* 成立状态；
* 查看详情；
* 加入或退出。

## P05 局详情

展示：

* 局名称、类型、场景和描述；
* 当前人数、最低成局人数、截止时间；
* 攒攒的群体观察；
* 成员昵称、摘要、意图和可提供价值；
* 隐私保护提示；
* 问攒攒是否适合；
* 加入或退出。

## P06 我的局

筛选：

* 全部；
* 我创建的；
* 等待成立；
* 已成立。

展示用户作为发起人或参与者加入的全部局。

## P07 我的资料与意图

### 资料

展示并允许编辑：

* 城市；
* 当前身份；
* 能力或兴趣；
* 可以提供；
* 一句话简介。

### 当前意图

展示：

* 状态：待确认、匹配中、已暂停；
* 标题和摘要；
* 目标；
* 可以提供；
* 有效期；
* 暂停或恢复匹配。

## P08 管理后台

包含：

* 数据概览；
* 内测码管理；
* 用户列表；
* Agent 日志。

## 7.9 视觉与交互规范

视觉方向参考 SayBriefly 的友好编辑感，但不复制其品牌资产。

### 色彩

| 用途 | 色值 |

# 八、Agent 核心能力

## 8.1 开放式对话理解

Agent 必须：

* 分析全部用户消息，不只看最后一句；
* 将长表达拆分为多个有效线索；
* 动态识别场景；
* 增量更新 Profile；
* 维护理解完整度；
* 只追问最关键的缺失信息；
* 在信息足够时生成 Intent 草稿。

当前 Demo 使用 Vercel AI SDK `ToolLoopAgent` 编排对话和工具调用，底层模型为阶跃星辰开放平台。模型只负责结合对话上下文生成自然回复、理解进度、结构化 Intent 草稿，以及在合适时生成 `ask_user_question` 表单参数；Intent 确认、加入、退出和创建局等业务动作仍由产品代码校验和执行，模型不能直接替用户改变业务状态。

### 8.1.1 轻量提问表单

Agent 可以在用户表达很短、方向模糊，或用户明确希望通过选项回答时调用 `ask_user_question`。使用规则：

* 每轮最多调用一次，每张表单最多 3 个问题；
* 支持单选、多选和一句话简答，选项一般包含“还不确定 / 都可以”或允许“其他”；
* 优先询问关系目标、开始方式、投入方式、城市范围和关键边界；
* 不得把固定五题问卷重新包装成表单，也不得要求用户选择内部场景标签；
* 信息已经足够生成 Intent 时不得发表单；
* 用户提交表单后的下一轮不连续调用表单，优先吸收答案生成可修订草稿，或用一句自然语言补问剩余关键缺口；
* 表单答案必须使用用户看到的标签回传上下文，不得展示 `side_project` 等内部值。

## 8.2 场景识别

内部识别为：

* `startup`：合作、合伙、招聘、加入团队、项目、客户、技术、销售等；
* `love`：恋爱、长期关系、伴侣、约会、认真认识、交友、兴趣活动等。

如果信号不明确，继续用通用问题理解，不要求用户自己选择技术标签。

## 8.3 搜索已有局

先取统一空间内所有未结束局，再按当前 Intent 场景优先排序，同时考虑：

* 是否已加入；
* 是否已满员；
* 目标相容性；
* 用户能否补足角色；
* 参与方式是否匹配。

## 8.4 创建局

没有合适结果时，Agent 可根据当前 Intent 创建：

* 工作与共创：一对一、招聘、试合作或小型多人局；
* 关系与交友：一对一或轻量多人局。

创建后，发起用户自动成为 host 和第一位成员。

## 8.5 推荐解释

解释必须包含：

* 为什么目标相容；
* 用户可能获得什么；
* 用户能提供什么；
* 当前仍有哪些不确定性；
* 不替用户做加入决定。

# 十三、技术方案

## 13.1 总体架构

```text
React 19 / Next.js 16 客户端
→ Next.js 服务端 Route Handler
→ /api/chat 对话理解接口
→ Vercel AI SDK ToolLoopAgent
→ OpenAI Compatible Provider
→ 阶跃星辰开放平台 Chat Completions API

当前网页 Demo 业务状态
→ 浏览器 LocalStorage（仅用于单机演示）

生产化目标
→ /api/app 统一业务接口
→ Cloudflare Worker 运行时
→ D1 数据库
→ Sites 版本保存与部署
```

## 13.2 前端

* React 19；
* Next.js 16 App Router 结构；
* 当前 Demo 使用 Next.js / Turbopack 构建，后续 Cloudflare 部署可评估 Vinext 兼容方案；
* 响应式桌面和移动布局；
* 桌面左侧导航，移动端底部导航；
* 对话区独立滚动，输入区不覆盖消息；
* `ask_user_question` 由聊天消息内的原生表单卡片渲染，支持桌面双列、移动端单列、键盘操作、必填校验和提交结果回显；
* 无第三方 UI 框架依赖；
* 当前 Demo 使用 LocalStorage 保存账号资料、消息、Intent 和参与状态，刷新页面后可继续体验；
* LocalStorage 不作为生产数据方案，也不保存模型密钥。

## 13.3 后端

当前 Demo：

* `/api/chat` Route Handler 处理 Agent 对话理解；
* 服务端使用 Vercel AI SDK `ToolLoopAgent` 编排模型与工具，使用 `@ai-sdk/openai-compatible` 连接阶跃星辰；
* 服务端读取模型环境变量并代理请求，客户端只获得整理后的回复、进度、Intent 草稿或经过 Schema 校验的表单参数；
* 模型输出必须经过 JSON 提取、字段校验、长度限制和场景枚举校验；
* `ask_user_question` 使用 Zod Schema 限制标题、问题数量、问题类型和选项数量；工具没有服务端业务执行函数，只返回待用户操作的表单；
* 单次模型请求超时为 15 秒；对超时、限流、5xx、网络错误和异常 JSON 自动重试一次，仍失败时使用本地规则降级；
* 模型请求期间允许切换页面，请求在后台继续；结果只更新聊天状态，不在“发现新局”等其他页面弹出模型提示；
* 只有用户主动“重新聊”或退出登录时，才取消尚未完成的聊天请求；
* 加入、退出、确认 Intent 和创建局均由确定性前端业务逻辑执行，不接受模型直接写入。

生产化目标：

* 单一 `/api/app` 路由处理业务动作；
* HttpOnly Cookie 会话；
* D1 持久化；
* 每次动作后返回最新 Bootstrap 数据；
* Agent 规则和所有写操作在服务端执行。

## 13.4 模型服务：阶跃星辰开放平台

| 配置 | 当前约定 |
| --- | --- |
| Provider | 阶跃星辰开放平台（StepFun） |
| Agent 编排 | Vercel AI SDK `ToolLoopAgent` |
| Provider 适配 | `@ai-sdk/openai-compatible` |
| API 协议 | OpenAI 兼容 Chat Completions |
| Base URL | `https://api.stepfun.com/v1` |
| 请求地址 | `POST /chat/completions` |
| 默认模型 | `step-3.5-flash` |
| 密钥变量 | `STEP_API_KEY` |
| 可选变量 | `STEP_API_BASE_URL`、`STEP_MODEL` |
| 超时 | 单次 15 秒，最多两次尝试 |
| 重试 | 超时、限流、5xx、网络错误和异常 JSON 重试一次；鉴权等不可恢复错误不重试 |
| 降级 | 本地规则式意图理解，仅在聊天输入区标记“模型繁忙 · 本地完成” |

实现约束：

1. 真实 API Key 只放在部署平台 Secret 或本机 `.env.local`，不得写入源码、PRD、客户端变量、截图和日志；
2. 禁止使用 `NEXT_PUBLIC_` 前缀暴露模型密钥；
3. 默认请求使用 `step-3.5-flash`，允许通过 `STEP_MODEL` 切换账号已开通的文本模型；
4. 发送给模型的上下文仅包含完成当前理解所需的最近对话，不发送邮箱、内测码和浏览器存储快照；
5. 普通模型结构化输出字段为 `reply`、`progress`、`intentDraft`；工具路径额外返回 `questionForm`；服务端校验通过后才返回客户端；
6. 上游错误不得向客户端透传鉴权信息、完整响应或服务端堆栈；
7. 正式上线前补充请求频控、审计日志脱敏、内容安全和第三方模型数据处理告知。

官方接入依据：

* [阶跃星辰 Chat Completion API](https://platform.stepfun.com/docs/zh/api-reference/chat/chat-completion-create)；
* [从 OpenAI 迁移至阶跃星辰](https://platform.stepfun.com/docs/zh/guides/developer/openai)；
* [Vercel AI SDK ToolLoopAgent](https://ai-sdk.dev/docs/reference/ai-sdk-core/tool-loop-agent)；
* [Vercel AI SDK OpenAI Compatible Provider](https://ai-sdk.dev/providers/openai-compatible-providers)。

## 13.5 `ask_user_question` 工具协议

服务端工具输入：

* `title`、可选 `description`、`progress`；
* `questions`：1–3 项，每项包含稳定 `id`、用户可读 `label` 和 `type`；
* `type`：`single_choice`、`multi_choice` 或 `short_text`；
* 选择题含 2–5 个 `options`，可配置 `required`、`allowOther`；
* 可选 `submitLabel`。

客户端状态：`pending → submitted`。提交前必须完成必填校验；提交后保留只读答案摘要，答案以中文标签序列化为“我通过快速表单补充了”消息，继续进入同一对话历史。

# 十七、隐私与安全原则

1. 对外只展示生成后的成员摘要，不展示原始对话；
2. 邮箱和敏感联系方式不展示给其他用户；
3. 用户加入前必须主动点击确认；
4. 重新聊只清理当前对话和意图，不误删已加入的局；
5. 所有修改和删除操作限定当前用户数据；
6. 内测码校验使用原子更新；
7. 会话 Cookie 使用 HttpOnly 和 SameSite；
8. 正式版本必须增加频率限制、日志脱敏和内容安全策略。
9. 模型 API Key 只允许存在于服务端 Secret，不进入客户端包、业务数据或 Agent 日志。
10. 用户对话会被发送给第三方模型服务用于当次理解，正式上线前必须在隐私说明中明确数据处理范围。

# 十九、MVP 验收标准

## 注册

* 用户无需选择创业或恋爱 Channel；
* 合法内测码和验证码可以完成注册；
* 重复邮箱、无效码、用尽码、停用码和过期码有明确错误提示；
* 登录页清楚说明合作、招聘、加入团队和恋爱交友均可使用。

## Agent

* 首问为开放式问题；
* 不出现固定五题问卷；
* 一条长消息包含足够信息时可直接生成 Intent；
* 信息不足时自然语言每次只追问一个关键问题；适合结构化选择时可展示不超过 3 题的轻量表单；
* 能识别工作和关系类意图；
* 能生成 Profile 和 Intent；
* 配置 `STEP_API_KEY` 后，对话响应来自阶跃星辰模型，客户端无法读取密钥；
* Agent 可在合适时调用 `ask_user_question`，网页正确渲染单选、多选、简答和“其他”；
* 表单必填校验、提交、只读回显和中文答案续聊正常，且不会在提交后连续发表单；
* 未配置密钥或有限重试后模型仍异常时，聊天输入区明确显示“模型繁忙 · 本地完成”，主流程仍可继续；
* 模型请求过程中切换到“发现新局”等页面不会弹出跨页面提示，返回聊天后消息和 Intent 结果仍然保留；
* 用户可继续补充并生成修订版；
* 用户确认后 Intent 进入 active；
* 重新聊保留已加入的局。

## 局

* 发现页同时展示工作与关系类开放局；
* 场景筛选可用；
* 查看详情可用；
* 加入和退出可用；
* 满员时不能加入；
* Agent 可创建新局；
* 用户创建或加入的局出现在“我的局”。

## 交互

* 左上场景入口能打开菜单并带入开场文本；
* 右下账号入口能打开个人资料、回到对话和退出登录；
* 对话消息不被顶部和输入框遮挡；
* 只有对话区滚动，不推动整个页面异常跳动；
* 桌面端和移动端主要流程可用；
* 页面不出现机器人脸、星芒或 emoji 产品图标。

## 后台

* 管理员可查看数据概览；
* 可创建、启用和停用内测码；
* 可查看用户列表；
* 可查看 Agent 日志；
* 普通用户不能进入管理后台。

# 二十、当前版本最终闭环

```text
用户从营销页理解：任何关系需求都可以聊
→ 使用邀请制邮箱账号进入
→ 和攒攒自然对话，不先选频道、不填问卷
→ 攒攒动态理解目标、处境、价值、投入和边界
→ 生成一版可修改 Intent
→ 用户确认 Intent
→ 攒攒从统一空间推荐合适的局
→ 用户查看理由和详情
→ 用户主动加入，或让攒攒创建新局
→ 用户在“我的局”和“我的”中持续管理
→ 后续对话继续校准推荐
```

这一闭环是当前邀请制 Demo 的核心产品边界。
